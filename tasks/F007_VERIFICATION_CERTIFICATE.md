# F007_VERIFICATION_CERTIFICATE

A redaction is only trustworthy if you can prove it worked — the single differentiator free/manual tools (PDF Gear) and Western auto-redactors (PII Buddy) do not offer, and the artifact a CA needs when sharing a client document under DPDP. Add `engine/verification.py`: after the renderer writes the output, re-parse it with `PDFParser`, re-run the detectors (via the shared `build_registry`), and assert (a) none of the redacted literal values are still extractable from the output text layer — checked both raw and whitespace-flattened, case-insensitive, so a normalised value like `226251716424` is caught even if retained spaced — and (b) no detector re-fires on a value that was supposed to be gone. Emit a `Certificate` (Pydantic): input and output SHA-256 (reuse `engine/decision.py::file_sha256`), redaction counts by type, timestamp, tool version, and PASS/FAIL. Critically, the certificate carries only counts, types, and pass/fail — never the raw PII values — so it is itself safe to share. Write it as a JSON manifest plus a one-page PDF (`render_certificate_pdf`). Expose `--certificate` on `redact` (and `redact-batch`, F006); a FAILED verification exits non-zero. Highlight-only redaction is non-destructive, so skip the certificate with a note. Scope note: verification covers the text layer; scanned/image redaction blanks pixels, for which "no extractable text" is the guarantee. Acceptance: a clean redaction verifies PASS with zero leaks; verifying the original (unredacted) file against the same detections FAILs with reported leaked types; the certificate JSON contains none of the redacted values; the PDF certificate is a valid one-pager.
