# F005_NAME_DETECTION

Add a NAME detector — the most common manual mark has no automated counterpart. Deterministic before AI, per engine rules: derive candidate names from evidence already in the document rather than a language model. Sources, in confidence order: (1) email local parts split on separators (`suraj.songara` → "suraj", "songara"), (2) LinkedIn/GitHub/UPI handle slugs (`linkedin.com/in/surajsongara`, camel-case splits like `SurajSongara`), (3) the layout analyzer's title/header line on page 1 when it is 2-4 capitalised tokens (resume convention). Candidates from two independent sources promote to high confidence; single-source candidates land in the ask band. Match candidates document-wide case-insensitively (including inside longer tokens, e.g. filenames) and emit one Detection per instance with the evidence source in `reasons`. Register at the `heuristic` fusion tier so regex detectors still win overlaps; an optional NER adapter (priority tier `ner=2`, optional extra, offline model) can come later behind the same detection type. Acceptance: on the resume TestFiles, first and last name instances are detected on every page with zero manual marks, no NAME false positives on the bank statements benchmark, and P004 precision numbers for existing types are unchanged.
